# 03 · 强制规则（Master Prompt · 不可绕过）

> 本文件是 SOP 的"宪法"
> 任何脚本 / 任何 AI 助手 / 任何人 都不得违反

---

## 6 条强制规则

### 规则 1：不得删减任何正文行
- 源文件里的每一行都必须出现在 render_prompt 里
- 包括看似冗余的引导句 / 注释句 / 金句
- 唯一例外：明显的工程包装层（`<!-- -->` 注释块 / 元数据标签块） · 且必须在 diff_report 上报

### 规则 2：不得压缩列表为摘要
- 5 个列表项 · render_prompt 里也必须是 5 个
- 不得用"等"字代替
- 不得用"主要包括 A、B、C 等"代替原列表

### 规则 3：不得省略任何数字 / 术语 / 标签 / 卡片标题
- 数字 100% 保留：90 天 / D+0 / D+360 / Phase 1 / 14 道护栏 / 5×5 矩阵
- 术语 100% 保留：Alem AI / Alem LLM / The JV / DeepSeek
- 标签 100% 保留：所有 chip / badge 文字
- 卡片标题 100% 保留：所有 H2 / H3 / **加粗短语**

### 规则 4：不得做术语替换
- 默认不替换
- 如本项目存在禁词清单 · 必须先获用户授权
- 替换时必须输出 `term_replacements: [{from: ..., to: ...}]` 到 diff_report.json
- 用户能从报告里一眼看到所有替换记录

### 规则 5：不得截断
- 不得自定 `max_chars` 上限
- 不得用"模型有 XXX 字符上限"这种非官方说法做截断
- 如真的超模型上限：**raise FAIL · 上报用户决策**
- 用户可选择：拆图 / 降低密度 / 改走矢量路径 / 接受风险

### 规则 6：任何 transform 必须在 diff_report.json 中显式记录
- 记录字段详见 `05_diff_report_schema.json`
- 缺审计报告 = 工程失败 · 必须重做

---

## 失败处置矩阵

| 失败情形 | 处置 |
|---|---|
| render_prompt 超模型上限 | raise FAIL · 上报用户 · 不得自行截断 |
| 工具调用失败 3 次 | 按 HL-2 停止重试 · 上报用户 |
| 出图后 OCR 检测漏字率 > 5% | 标记 `density_risk: high` · 提示降级到矢量路径 |
| diff_report.json 未产出 | 整个项目标记 FAIL · 不得交付 |

---

## 违规检测

### 如何判断"我（AI）刚刚是否违规"
在每次工具调用前自检：

- [ ] 我是不是要写一个 `max_chars` 字符上限？→ ❌ 违反规则 5
- [ ] 我是不是要"先提取要点再发给模型"？→ ❌ 违反规则 1
- [ ] 我是不是要做术语替换但没准备 diff_report？→ ❌ 违反规则 4 + 规则 6
- [ ] 我是不是要把列表压成一句话？→ ❌ 违反规则 2
- [ ] 我是不是要省略小数字（"约 3 天"代替"3 天"）？→ ❌ 违反规则 3

任何一项命中 → 立即停手 · 不执行 · 上报用户。
